# AGENTS.md

## Project Status

项目仍处于玩法和架构快速调整阶段，目录结构、模块边界、命名和职责划分都可能继续变化。

本文件的目标不是冻结当前架构，而是约束 AI 在频繁修改中保持安全、可读、可验证：

* Cocos 场景、Prefab、Meta、Inspector 引用相关规则必须严格遵守。
* 目录结构、模块边界、命名规则作为当前方向，不作为不可变限制。
* 如果现有结构不适合当前需求，可以提出调整方案，但必须说明改动范围、原因和验证方式。
* 优先小步收敛，不要为了追求“架构正确”而做大范围无关重构。

## Rule Priority

AI 修改代码时，规则优先级如下：

1. 当前用户明确需求
2. `AGENTS.md`
4. 当前代码现状

如果 `AGENTS.md` 与实际代码明显冲突，不要强行按文档改代码；先指出冲突，并优先做最小、安全、可验证的修改。

## Tech Stack

* Cocos Creator `3.8.8`
* TypeScript
* `npm`
* Node 内置测试框架 `node:test`

## Project Scope

这是一个 Cocos Creator `3.8.8` + TypeScript 的 2D 棋盘弹球项目。

当前重点围绕棋盘系统展开，包括：

* 5×5 棋盘
* 可放置 Entity
* 武器与转向器
* 弹球移动与路径预测
* 武器尾巴触发充能
* 商店商品拖拽放置
* 同类同级实体合并

不要擅自引入尚未确认的大系统，例如敌人系统、关卡系统、背包系统、战斗结算系统、存档系统等。
如果确实需要引入新系统，必须先说明它和当前棋盘系统的关系。

## Assets Structure

`assets` 目录大致分层如下：

* `scenes/`：Cocos 场景资源。
* `scripts/`：TypeScript 脚本。
* `scripts/board-bootstrap/`：场景装配、动画驱动、展示刷新、拖拽桥接。
* `scripts/board-renderer/`：棋盘与实体视觉渲染。
* `scripts/board-runtime/`：棋盘运行时、实体存储、step 结算、事件派发、武器充能。
* `resources/`：运行时动态加载资源入口。

本节只是当前结构参考，不是强制结构。
修改前必须以实际目录、代码引用和场景挂载为准。

如果现有目录不适合当前需求，可以提出调整方案，但不要在没有说明影响范围的情况下直接大规模移动文件。

## Terminology

优先沿用以下主命名：

* `Board`：棋盘整体。
* `Cell`：棋盘单个格子。
* `Entity`：可放置在棋盘上的实体总称。
* `Weapon`：武器实体。
* `Redirector`：改变弹球方向的转向器。
* `Ball`：弹球。
* `Tail`：武器尾巴 / 充能触发器。
* `Charge`：武器充能。
* `ShopItem`：商店商品表现。
* `Placement`：放置行为。
* `Merge`：合并行为。
* `Runtime`：规则运行时。
* `Renderer`：视觉渲染层。
* `Bootstrap`：场景装配入口。

同一业务链路内尽量保持命名一致。
如果现有命名不足以表达新概念，可以新增命名，但必须说明它和已有概念的区别。
不要盲目做全局重命名，特别是涉及场景挂载、Inspector 字段、Prefab 引用时。

## Module Boundary Direction

以下是长期方向，不是当前阶段的绝对限制：

* `Runtime` 尽量保持为纯规则层，不直接依赖 Cocos 节点、Prefab、UI、颜色、动画、拖拽输入。
* `Renderer` 负责 Cocos 节点、Sprite、Label、颜色、布局和视觉刷新。
* `Bootstrap` 负责组装 Runtime、Renderer、输入桥接、动画驱动和调试入口。
* 商店展示逻辑尽量不要进入 Runtime。
* 武器具体行为不要过早塞进通用投射物或通用实体基类。
* Runtime 输出结果应尽量是业务事件或状态变化，不直接操作视觉节点。

如果当前代码还没完全做到，不要为了追求分层正确而大范围重构。
修改时优先不扩大耦合，有机会再小步收敛。

## Coding Rules

### 必须做

* 修改前先读相关文件，不允许只凭文件名猜。
* 优先做最小修改，不顺手重构无关代码。
* 保持现有命名风格：

  * 类名 PascalCase
  * 变量和函数 camelCase
  * 节点名用语义化英文
* 新增或修改代码时，补充必要注释，但禁止注释噪音：

  * 公共类、核心函数、关键业务流程、复杂判断、特殊边界必须写注释。
  * 注释说明业务意图、设计原因、约束来源或风险点。
  * 不要写只复述代码表面行为的注释。
  * 如果逻辑需要大量注释才能看懂，优先改善命名和职责。
* 每个文件、类、函数职责尽量单一，命名直接表达职责。
* 如果一个类同时承担两种业务职责，优先说明问题，再决定是否拆分。
* 以下情况优先带后缀说明真实类型：

  * `Node` 引用字段用 `xxxNode`
  * 世界坐标字段用 `xxxWorldPos`
  * 运行时上下文字段用 `xxxContext`
* 如果旧字段名、旧接口名、旧配置名不符合当前主命名：

  * 仅当它位于本次修改的直接链路内，且范围小、风险低、可验证时，才允许同步收敛。
  * 如果会影响多个模块、场景挂载、序列化字段、Prefab 引用，必须先说明影响范围，不要直接改。

### 禁止做

* 不要删除看起来无用但未确认用途的代码。
* 不要新增“看起来通用，实际绑死某个武器 / 功能”的抽象。
* 不要只改名字，不处理内部混杂职责。
* 不要为了抽象而抽象。
* 不要把调试逻辑和正式业务逻辑长期放在同一个组件里。
* 不要把武器专属逻辑过早放进通用投射物基类。
* 不要把配置层概念偷偷塞进运行时行为类。
* 不要在新代码里为同一概念随意引入第二套别名。
* 不要新增模糊命名：

  * `data`
  * `info`，除非它本身是明确模型，如 `DamageInfo`
  * `manager`，除非它真的是管理器职责
  * `utils`，除非是明确的纯工具模块

## Config / Runtime Boundary

* 配置层只描述静态定义，例如类型、等级、价格、初始方向、伤害、充能需求。
* 运行时行为层负责移动、碰撞、合并、充能、触发、事件派发。
* 不要让配置对象承担运行时状态。
* 不要让运行时类反向依赖商店展示字段。
* 不要新增“看起来像规则但不参与运行”的描述字段；规则定义必须被实际运行链路读取，否则应放入文档说明而不是配置对象。

这是长期方向。
如果当前代码存在混用，修改时不要继续扩大混用；可以在本次相关链路内小步收敛。

## Scene / Asset Rules

* 修改前必须确认节点是“场景静态节点”还是“代码动态创建节点”。
* 不要随意改资源路径、场景文件名、脚本文件名、`.meta` 文件。
* 修改 `.scene` / `.prefab` / `.meta` 前，必须确认：

  * 当前改动是在改字段，还是改节点结构
  * 是否有场景脚本挂载依赖
  * 是否有 Inspector 拖拽引用
  * 是否有资源 UUID / `__id__` 关联
* 能不改场景结构，就不要改场景结构。
* 如果必须改场景结构，优先做最小追加式修改，不要随意重排已有序列化块。
* 不要为了修复引用删除 `.meta` 重建。
* 移动或重命名资源时，优先保留原 `.meta`，避免 UUID 变化导致引用断裂。
* 除非明确要求，不要手写或大规模重排 `.scene` / `.prefab` 的复杂结构。

## Cocos Serialization Rules

* 不要随意重命名带有 `@property`、被 Inspector 绑定、被场景序列化引用的字段。
* 修改组件字段前，必须判断该字段是否可能被 `.scene` / `.prefab` 引用。
* 如果必须重命名序列化字段，需要说明迁移影响，并检查相关场景或 Prefab。
* 不要通过删除 `.meta`、重建资源、重排序列化块来修复引用问题。
* 涉及场景、Prefab、Inspector 绑定的修改，必须比普通 TypeScript 修改更谨慎。

## Debug Rules

* 调试入口优先放在 `*Debug*`、`*-debug-*`、`board-debug-api.ts` 等明确调试命名的文件中。
* 正式运行时模块不得依赖调试模块。
* 临时调试代码完成任务后应删除，除非它本身是明确的调试工具。
* 不要让临时调试按钮、全局变量、控制台入口长期混入正式业务流程。

## Collaboration Rules

除非是非常小的单点修改，否则 AI 动手前需要说明：

* 理解到的需求
* 第一轮要看的文件或链路
* 预计修改范围

如果改动较大，先给修改方案或修改范围，再落代码。

不要只因为“现在能跑”就保留明显错误的命名或职责混杂。
但也不要为了追求“架构正确”而做大范围无关重构。

## After Editing

1. 再读一遍受影响文件，确认没有顺手改出额外行为。
2. 运行能确认的最低检查：

   * 优先运行 `npm test`
   * 如果 `npm test` 不存在，或失败原因与本次改动无关，说明原因，不要为了通过检查随意修改测试脚本
   * 可补充运行项目已有的 TypeScript 检查、构建命令或最小验证命令
3. 如果新增或删除脚本文件，确认 `.meta` 一并正确处理。
4. 如果改了场景 / UI / 资源，检查是否会出现：

   * 重复节点
   * 丢失脚本挂载
   * 丢失 Inspector 引用
   * `.meta` / UUID 关联断裂
5. 提交说明里写清楚：

   * 改了哪一层
   * 是否动了场景 / 资源
   * 跑了什么验证
   * 哪些信息仍未确认
